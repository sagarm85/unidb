# unidb Engine Architecture & Product Guide — context & regeneration

This file holds the provenance and regeneration notes for
[`unidb_engine_architecture.html`](unidb_engine_architecture.html) (and the
generated `unidb_engine_architecture.pdf`). It is kept **out** of the guide
itself so the guide reads as an end-user document with no internal
milestone/version bookkeeping on its first page. Keep this file up to date when
you edit the guide.

## What the guide is

An **end-user-facing** architecture & product guide for unidb: what the engine
is, how it stores and protects data, how transactions and concurrency behave,
the SQL layer (types + query examples), search/indexing/graph, the event
stream, the full REST API reference, operations/HA, a configuration &
performance-tuning reference (§11 — every overridable engine/server setting,
its default, purpose, and measured performance impact), measured performance
vs Postgres, correctness/testing, known limitations, roadmap, and (§15) an
alphabetical glossary of every abbreviation used in the guide (ACID, MVCC,
WAL, CDC, JWT, RLS, etc.) in book-appendix style.

When a new section introduces an abbreviation not already in §15, add it there
too — keep the glossary exhaustive over the guide's own vocabulary, not a fixed
list.

Audience: prospective and current **users** (application developers, technical
evaluators) — not unidb engine engineers. Accordingly the guide avoids internal
implementation identifiers: source file/module names, function names, WAL record
kind names, and internal milestone / phase / decision / item codes
(e.g. M0–M11, Phases 1–6, P1.a, D1–D13, item 11, R1–R4) are intentionally kept
out. When a feature is added, describe it as a capability/behavior, not by its
internal tracking id.

## Coverage snapshot (for internal reference)

As of the last regeneration, the guide reflects the engine through: storage
core, MVCC + SQL, durable secondary indexes (B-tree, on-disk HNSW graph vector,
full-text, edge adjacency), graph, event queue + realtime dispatcher, REST
server with transaction sessions / cursors / batch, catalog introspection
(`information_schema`), `JOIN … USING`, observability metrics, structured logs,
concurrent SQL writes, parallel scan, autovacuum (now per-table, with a
cost-based I/O throttle), object storage (S3/MinIO tiering over engine
metadata), an indexed + push-based event queue (flat poll latency at any
table size, sub-millisecond idle-subscriber delivery), change-data-capture
(before/after row images, Debezium- and Supabase-compatible wire formats,
per-consumer lag observability), a multi-page catalog (no practical schema-size
ceiling), time-based point-in-time restore, row-level logical replication into
another database, and Phase-6 ops/HA (segmented WAL, replication slots, read
replicas + failover, backups + PITR, users/roles/JWT, TLS, audit log). Backlog
item 24 (authz v2 policies) is the one shipped-tracker item not yet started;
§13's "Known limitations" and §14's roadmap reflect that. §6.2 also carries a
short worked example — "Embedding search over stored files" — showing that
semantic search over object-storage content is an application-level pattern
(a companion `VECTOR(n)` table + `NEAR()`, joined back to object metadata),
not an engine or `unidb-storage` crate feature; there is no automatic
file-content embedding pipeline.

Section 13 ("Known limitations, bugs & roadmap priority") is regenerated by
cross-checking `docs/backlog/backlog_index.md` (shipped vs. not-started) and
`docs/backlog/19_sql_surface_gaps.md` for currently open, user-visible
behavior bugs (as opposed to missing features) — e.g. gap **G10** (row-path
`IS NULL`/`LIKE` predicate parity, esp. under `NEAR`) is the one open bug as of
this writing. Historical bugs found-and-fixed during development (e.g. the
transaction-abort-ordering issue under concurrency, tracked internally as
backlog item 16) are **not** listed there — they are fixed, permanently
regression-tested, and mentioned only as a trust signal in §12/§13.1, not as
open items. When refreshing, re-grep `MEMORY.md`/`PROGRESS.md`/backlog docs for
new "bug" mentions before assuming §13.1 is still complete. The event/CDC
row-level-security gap (§13.2) was surfaced by the CDC work itself (its own
"known limitations" note) — cross-check each newly shipped item's own
known-limitations list, not just the top-level backlog registry, when
refreshing.

## Regeneration (produces page-numbered PDF)

The PDF is rendered from the HTML by headless Google Chrome driven over the
DevTools protocol, so a **page number** can be placed in the footer (Chrome's
print CSS cannot do page-counter margin boxes on its own). Run:

```bash
cd docs/design
node render_pdf.mjs unidb_engine_architecture.html unidb_engine_architecture.pdf
```

`render_pdf.mjs` launches Chrome headless, prints to PDF with
`displayHeaderFooter` enabled (empty header, page-number footer), and writes the
result. It needs only Google Chrome and Node ≥ 22 (for the built-in
`WebSocket`); no npm install.

Workflow: **edit the HTML, then re-run the command above.** The first page is
the title + contents (no full-page cover); the table of contents is clickable
(internal `#anchor` links); every content page carries a footer page number.
