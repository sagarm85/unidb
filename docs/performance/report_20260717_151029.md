## What this measures (self-contained — no other docs needed)

The **W0→W4 decomposition ladder**: one durable transaction, adding one data
model at a time, so the multi-model write tax is read by subtraction. Every
commit is one group-coalesced `fsync` (matched durability). The number
reported is **marginal ms per durable commit** at each pre-grown table size
(`MM_SAMPLE=200` commits averaged; pre-grow not timed).

| Rung | What the single transaction does |
|------|----------------------------------|
| **W0** | plain relational row INSERT (WAL + heap + fsync) |
| **W1** | W0 + a B-tree secondary index entry |
| **W2** | W1 + a `VECTOR(128)` value + its ANN index (HNSW) |
| **W3** | W2 + a graph edge + adjacency-index maintenance |
| **W4** | W3 + event-queue capture — **the full four-model commit** |

`W4−W0` is the total multi-model tax; `W4/W0` is the multiplier. The thesis:
one shared fsync makes W4 ≈ W0 (not N×), and it should stay that way as rows
grow. A rising `W4/W0` means per-model index CPU (esp. HNSW) is eroding it.

## Table 4 — unidb multi-model (1 atomic txn) vs the replaced stack, at scale

The §6 headline. unidb commits **four model-writes in ONE atomic transaction**
(relational + `VECTOR(128)`+HNSW + graph edge + event) — **1 `fsync`, all-or-
nothing**. The **replaced stack** does the *same four writes* as **four
independent systems with no shared transaction** (Postgres row + pgvector
+HNSW + a graph adjacency table + an outbox queue), each its own connection and
its own durable commit — **4 `fsync`s, 4 round-trips, and NO cross-system
atomicity** (a crash mid-sequence leaves a torn record; see the
crash-consistency test). The `PG relational only` column is the stack's
single-model *floor* (one write), kept for reference — it is **not** the
baseline. This is a **conservative** proxy: real Neo4j/Kafka/Qdrant are heavier
than PG tables, so the true stack tax is larger.

_Durability lens: unidb `fsync`, Postgres `wal_sync_method=fsync` (matched);
single writer, so no group-commit coalescing on either side._

| txns | unidb txns/s | unidb ms/txn | stack (4-sys) txns/s | stack ms/txn | **unidb ÷ stack** | PG relational only txns/s |
|-----:|-------------:|-------------:|---------------------:|-------------:|:-----------------:|--------------------------:|
| 1000 | 70 | 14.336 | 1079 | 0.927 | **0.06×** | 7386 |
| 10000 | 60 | 16.756 | 1077 | 0.928 | **0.06×** | 7384 |
| 100000 | 38 | 26.543 | 976 | 1.024 | **0.04×** | 7363 |

`unidb ÷ stack > 1` means unidb's single atomic commit beats the four-system
dual-write on throughput. **The throughput edge is `fsync`-cost-dependent:**
unidb pays 1 durable sync per record, the stack 4 — but that advantage only
dominates when a durable sync is *expensive*. Under Docker's shared-VM `fsync`
(cheap, buffered — see the durability caveat) the per-model HNSW/index CPU
paid on **both** sides dominates instead, so the ratio sits near parity and
the win narrows; on a native host with a real flush-to-platter sync the
4→1 collapse is worth far more. The **unconditional** win is the crash-
consistency below — one atomic commit vs four with no shared transaction —
which no `fsync` setting changes.

**Crash-consistency:** the replaced stack recovered a **torn record** — the
relational row is durably present while its embedding/edge/event are absent
(no transaction spans the four systems to undo it). unidb recovers **0
orphans** by construction — a crash before `WAL_TXN_COMMIT` undoes all four,
proven in `tests/crash` (`item16_incomplete_four_model_txn_leaves_zero_orphans`,
`item16_committed_four_model_txn_survives_intact`). No fsync setting buys the
stack this.

## Caveats

- Single node, `fsync` commit sync, group-commit on. The per-commit floor
is this storage's fsync latency.
- Sizes swept: `[1000, 10000, 100000]` (`MM_SIZES` to override — e.g. millions). W2–W4 build
the ANN/graph indexes synchronously, so large pre-grows are slow **by design**
— that cost is exactly what the async-derivation design (parked) would move off
the commit path if `W4/W0` is seen rising.
- Marginal-commit sample = 200 (`MM_SAMPLE`); numbers carry a few-percent
noise, the *trend across sizes* is the signal.
- Table 5's order count = 20000 (`MM_FK_ORDERS` to override); its FK
check is single-column, point-lookup (item 35's implicit unique index) —
a composite or non-indexable FK column falls back to an O(n) heap scan on
unidb (documented limitation, not exercised by this table).
