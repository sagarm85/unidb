## What this measures (self-contained — no other docs needed)

The **W0→W4 decomposition ladder**: one durable transaction, adding one data
model at a time, so the multi-model write tax is read by subtraction. Every
commit is one group-coalesced `F_FULLFSYNC` (matched durability). The number
reported is **marginal ms per durable commit** at each pre-grown table size
(`MM_SAMPLE=50` commits averaged; pre-grow not timed).

| Rung | What the single transaction does |
|------|----------------------------------|
| **W0** | plain relational row INSERT (WAL + heap + fsync) |
| **W1** | W0 + a B-tree secondary index entry |
| **W2** | W1 + a `VECTOR(128)` value + its ANN index (HNSW) |
| **W3** | W2 + a graph edge + adjacency-index maintenance |
| **W4** | W3 + event-queue capture — **the full four-model commit** |

`W4−W0` is the total multi-model tax; `W4/W0` is the multiplier. The thesis:
one shared fsync makes W4 ≈ W0 (not N×), and it should stay that way as rows
grow. A rising `W4/W0` means per-model index CPU (esp. HNSW) is eroding it.

## Table 3.1 — Bulk stress: insert + full-scan select at scale

Scaling behaviour of a single-table load and a full-table scan as the row count
climbs. For each size a **fresh** table is built, `n` rows are bulk-inserted
(batched prepared single-row inserts, one durable commit per 5000
rows — the same method on both engines), then a full-**heap** scan is timed
(`COUNT(*) WHERE body <> 'x'` — a predicate on the non-indexed `body` column, so
neither engine can serve it index-only; this measures real scan throughput, not a
count optimizer). Throughput is records/sec. Sizes swept: `[1000, 10000]` (`MM_BULK_SIZES`
to override — e.g. push to `5000000` or `10000000` for a heavier run). The default
tops out at 2M to keep a full report reasonable; the engine handles ≥10M.

**On the scan gap at scale:** Postgres runs a **parallel** sequential scan (multiple
worker processes) once the table crosses its parallel threshold, while unidb scans
**single-threaded**; so a large scan-side lead is Postgres's parallel-query capability,
not a per-row storage-speed difference (at small sizes, below PG's threshold, the two
are much closer).

## Caveats

- Single node, `F_FULLFSYNC` commit sync, group-commit on. The per-commit floor
is this storage's fsync latency.
- Sizes swept: `[100, 1000]` (`MM_SIZES` to override — e.g. millions). W2–W4 build
the ANN/graph indexes synchronously, so large pre-grows are slow **by design**
— that cost is exactly what the async-derivation design (parked) would move off
the commit path if `W4/W0` is seen rising.
- Marginal-commit sample = 50 (`MM_SAMPLE`); numbers carry a few-percent
noise, the *trend across sizes* is the signal.
- Table 5's order count = 1000 (`MM_FK_ORDERS` to override); its FK
check is single-column, point-lookup (item 35's implicit unique index) —
a composite or non-indexable FK column falls back to an O(n) heap scan on
unidb (documented limitation, not exercised by this table).
