## What this measures (self-contained — no other docs needed)

The **W0→W4 decomposition ladder**: one durable transaction, adding one data
model at a time, so the multi-model write tax is read by subtraction. Every
commit is one group-coalesced `fsync` (matched durability). The number
reported is **marginal ms per durable commit** at each pre-grown table size
(`MM_SAMPLE=200` commits averaged; pre-grow not timed).

| Rung | What the single transaction does |
|------|----------------------------------|
| **W0** | plain relational row INSERT (WAL + heap + fsync) |
| **W1** | W0 + a B-tree secondary index entry |
| **W2** | W1 + a `VECTOR(128)` value + its ANN index (HNSW) |
| **W3** | W2 + a graph edge + adjacency-index maintenance |
| **W4** | W3 + event-queue capture — **the full four-model commit** |

`W4−W0` is the total multi-model tax; `W4/W0` is the multiplier. The thesis:
one shared fsync makes W4 ≈ W0 (not N×), and it should stay that way as rows
grow. A rising `W4/W0` means per-model index CPU (esp. HNSW) is eroding it.

**No shared transaction.** A crash between any two commits leaves a torn record.
Contrast unidb W4: one `WAL_TXN_COMMIT`, one `fsync`, all-or-nothing.

**Conservative vs realistic:** item 17's conservative baseline simulates the
queue as a PG outbox table on the SAME server (no inter-process TCP). This
table adds Redpanda in a SEPARATE Docker container — the true replaced-stack
cost, including real inter-process TCP latency for the event leg.

**PG all-in-one:** a single PG transaction over all four model-writes (no
Redpanda) shows the best-case throughput for a PG-only deployment — one
fsync, four round-trips within the same server. This is faster than the
replaced-stack rows but still pays client-server round-trip overhead per
write that unidb's embedded path does not.

**Reading the table:**
- `unidb ÷ this > 1` = unidb is faster (embedded + 1 fsync wins).
- `unidb ÷ this < 1` = unidb is slower (the other system has lower overhead
for that workload size; honest — unidb is not a CRUD incumbent).
- The gap from `conservative` to `realistic` is the true inter-process
TCP tax of the Redpanda leg — the overhead the outbox proxy understates.
- The gap from `PG all-in-one` to `conservative` is the cost of splitting
the same four writes across four separate commits (no group-commit
coalescing between them).
- The atomicity column is the **unconditional** win: unidb guarantees
all-or-nothing regardless of `fsync` cost. No tuning buys the stack this.

## Caveats

- Single node, `fsync` commit sync, group-commit on. The per-commit floor
is this storage's fsync latency.
- Sizes swept: `[1000, 10000, 100000]` (`MM_SIZES` to override — e.g. millions). W2–W4 build
the ANN/graph indexes synchronously, so large pre-grows are slow **by design**
— that cost is exactly what the async-derivation design (parked) would move off
the commit path if `W4/W0` is seen rising.
- Marginal-commit sample = 200 (`MM_SAMPLE`); numbers carry a few-percent
noise, the *trend across sizes* is the signal.
- Table 5's order count = 20000 (`MM_FK_ORDERS` to override); its FK
check is single-column, point-lookup (item 35's implicit unique index) —
a composite or non-indexable FK column falls back to an O(n) heap scan on
unidb (documented limitation, not exercised by this table).
